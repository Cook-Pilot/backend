1. 뭘 왜 바꿨나

개인 레시피 버전의 조정 내용이 opaque JSONB(adjustment_payload) 하나에 뭉쳐 있어 "뭘 바꿨는지" 추적이 불가능했다. 이번 변경으로 조정값을 관계형 diff 테이블 2개로 옮기고, 인메모리로만 있던 개인 버전·리뷰 저장을 실제 PostgreSQL(JPA)로 전환했다.

대화에서 확정된 설계 결정

결정내용
표현 방식diff 행 (방식 A) — 변경분만 저장, 렌더링은 원본 + diff 합성. 스냅샷 복사 아님
diff 기준항상 원본 레시피 기준 누적 — 어떤 버전이든 원본 + 그 버전 diff만으로 렌더링 끝. 부모 체인 재생 없음
버전 이력v1, v2, v3… version_number로 쌓고 parent_version_id로 계보 기록. 최신 버전이 is_default
새 재료 문제ADD 타입은 원본 참조 없이 자기 데이터를 통째로 가짐. DB CHECK로 강제
단계 변경재료와 동일한 ADD/MODIFY/REMOVE. "흔든다→젓는다"도 MODIFY 한 건. 끼워넣기는 insert_after_step_index 앵커
payload 운명adjustment_payload 컬럼 삭제 — 입력은 정제된 diff로만 들어온다
리뷰 영속화포함 — 리뷰를 먼저 저장하고 버전이 source_review_id로 역참조 (추적 안 끊김)
API 경로신규 경로 설계는 보류 (사용자 결정). 기존 GET 2개만 응답 형태 갱신, 파생은 서비스 레이어까지 구현

CHECK: ADD 는 원본 참조 금지 + 필수값 보유, MODIFY/REMOVE 는 원본 참조 필수

그 외

  • ALTER TABLE personal_recipe_versions DROP COLUMN adjustment_payload — 현재 내용물이 전부 목데이터라 손실 없음
  • 시드: 목 유저(00000000-…-01) + 데모 레시피 2종(라면/김치볶음밥, 재료·단계 포함)을 인메모리 픽스처와 동일한 고정 UUID로 삽입 — 개인 버전/리뷰의 FK가 실제 행을 가리키게 하는 다리. 레시피 조회 API의 JPA 전환은 별도 이슈

3. 합성(렌더링) 규칙 — 소금/칵테일 시나리오

합성기는 DiffComposer(순수 함수, DB 무관). 원본 + diff → 완성된 내 레시피.

예: 칵테일 원본에 "2번 뒤 소금 추가" + "흔들기→젓기"

#합성 결과 단계origin
0얼음을 셰이커에 넣는다ORIGINAL
1재료를 붓는다ORIGINAL
2소금 한 꼬집을 넣는다ADDED (anchor=1)
310초간 흔든다 → 바 스푼으로 저어서 섞는다MODIFIED
  • MODIFY: null 필드는 원본 유지, non-null만 오버라이드 (수량만 바꾸기 / 행위 통째로 바꾸기 둘 다 커버)
  • ADD(단계): 앵커 뒤 삽입, 같은 앵커끼리는 sort_order, 인덱스는 0부터 재부여
  • REMOVE: 결과에서 제외. 제거된 단계 뒤에 앵커된 ADD는 살아남음 (앵커는 원본 인덱스 기준)
  • 범위 밖 앵커(원본 3개인데 anchor=99)는 맨 뒤에 붙음. -1은 맨 앞

파생 시맨틱

원본 ──diff──▶ v1 ──(부모 diff 복사 후 수정)──▶ v2 ──▶ v3 …
렌더링은 언제나: 원본 + 해당 버전 diff (체인 재생 없음)
파생 요청의 diff가 null이면 부모 diff 그대로 복사(제목/요약만 바꾼 파생)
